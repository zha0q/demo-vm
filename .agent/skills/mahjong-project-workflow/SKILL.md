# 麻将项目工作流

## 核心原则

从 `.agent/PLAN.md` 出发，把任务拆小，并始终保持规则逻辑、React UI 和 Three.js 渲染之间的分离。

## 任务开始流程

1. 阅读 `.agent/PLAN.md`。
2. 如果存在 `.agent/TASK.md`，就先阅读；如果用户要求做任务追踪，再创建它。
3. 查看 `git status --short`。
4. 判断当前该用哪个本地技能：
   - 规则：`mahjong-solitaire-rules`
   - IQ：`mahjong-in-game-iq`
   - Three.js：`threejs-mahjong-scene`
5. 在实现规则或计分前，先补测试或更新测试。

## 开发顺序

除非用户明确要求，优先按这个顺序推进：

1. 常量和配置。
2. 匹配规则和棋盘可选性。
3. 步骤队列。
4. 局内 IQ。
5. 撤销、提示、洗牌、可解性。
6. React 页面拆分。
7. Three.js 动画。
8. 更多菜单和自动完成。
9. 响应式 QA。

## 必需命令

每个任务期间都要跑聚焦测试：

```bash
npm test -- test/name.test.ts
```

在准备声称完成前，要跑完整验证：

```bash
npm test
npm run build
```

凡是会影响浏览器表现的改动，还要跑：

```bash
npm run dev
```

然后用手动或自动化浏览器方式验证对应视口。

## 知识维护

在最终回复之前，使用 `project-knowledge-maintenance`。

当实现改变了持久事实时，更新相关的 `knowledge/*.md`：

- 架构或模块边界变化。
- 玩法、IQ、计分、队列、提示、撤销、洗牌或可解性变化。
- 视觉令牌、布局、动效或页面设计变化。
- 测试命令、QA 要求或浏览器验证变化。
- 未来 Agent 需要理解的新决策。

当重复工作流或常见坑应该指导未来 Agent 时，更新 `.agent/skills/*/SKILL.md`。

## 任务追踪

如果使用 `.agent/TASK.md`：

- 只有已经验证过的工作才能移到 `已完成`。
- 未完成的工作留在 `待完成`。
- 新发现的任务要直接补进去，不要藏在叙述里。
- 没有亲自检查视口之前，不要把浏览器 QA 标成完成。

## 边界

- 不要复制 Vita Mahjong 的素材、原始布局、Logo、品牌文案或音效。
- 不要在本来只需要局部重构的时候，把当前原型整个推翻重做。
- 不要让 UI 或渲染层变成游戏真相的来源。
- 没有测试或与功能相称的浏览器检查，就不要说功能完成了。
- 如果代码、测试、设计或工作流发生变化，就不要跳过知识扫尾。
