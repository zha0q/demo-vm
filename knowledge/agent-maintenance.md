# 智能体维护知识

## 目标

随着实现演进，保持项目知识和本地技能同步更新。

## 需要更新什么

在以下情况更新 `knowledge/`：

- 架构边界发生变化。
- 新模块被加入。
- 玩法行为发生变化。
- 设计令牌或布局规则发生变化。
- 测试策略发生变化。
- 发现重复出现的坑或决策。

在以下情况更新 `.agent/skills/`：

- 未来 Agent 需要新的工作流护栏。
- 某个重复错误应该被预防。
- 某个领域特定检查清单已经稳定。
- 项目约定应该在未来请求中自动触发。

在以下情况更新 `.agent/PLAN.md`：

- 任务顺序发生变化。
- 计划中的模块被重命名或替换。
- 范围发生变化。
- 验收标准发生变化。

在以下情况更新 `.agent/Design.md`：

- 视觉方向发生变化。
- 引入新的页面模式。
- 组件令牌或布局规则发生变化。

## 任务结束知识扫尾

最终回复前，执行这轮检查：

1. 检查已变更文件。
2. 判断是否有已变更行为影响 `knowledge/*.md`。
3. 判断是否有重复工作流或常见坑影响 `.agent/skills`。
4. 更新最小范围的相关文档。
5. 对不显而易见的决策，在 `knowledge/decision-log.md` 添加带日期的记录。
6. 使用 `find`、`sed`、`grep` 或相关测试执行文档验证。
7. 说明更新了什么、没有更新什么。

## 半自动维护

除非工作流明确要求，或脚本/钩子强制执行，否则 Agent 无法真正自动更新文档。本项目同时使用两种方式：

- 软自动化：开发任务结束时必须使用 `project-knowledge-maintenance` 技能。
- 可选硬自动化：之后可以添加 git hook 或 npm 脚本，用来检查代码变更后是否缺少知识扫尾。

建议的未来脚本：

```bash
npm run knowledge:check
```

建议行为：

- 检测已变更的 `src/**` 或 `test/**` 文件。
- 如果没有 `knowledge/**`、`.agent/PLAN.md`、`.agent/Design.md` 或 `.agent/skills/**` 变更，则给出警告。
- 打印检查清单，而不是自动修改文档。

## 为什么不完全自动化

知识更新需要判断。脚本可以发现代码发生了变化，但无法可靠判断哪些架构或产品知识已经变成持久事实。更安全的模式是自动提醒加上显式 Agent 审查。

## 文件记录规则

修改任何文件时，遵循用户要求的 diff 记录规则：

- 内容变更前，用绝对路径调用 `beforeEditFile`。
- 修改文件。
- 变更完成后，用相同绝对路径调用 `afterEditFile`。
- 每个已变更文件保持一组 before/after 配对。
